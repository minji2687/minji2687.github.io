## 비트, 바이트

**비트(bit)** 는 0 또는 1 하나. **바이트(byte)** 는 비트 8개를 묶은 것으로, `00000000`~`11111111`(10진수로 0~255) 범위를 표현한다. 네트워크로 오는 데이터는 결국 이 바이트가 순서대로 나열된 것뿐이다 — `Uint8Array`나 `ArrayBuffer`가 딱 이 "바이트 나열"을 표현하는 자료구조다.

**"바이트 하나 = 스위치 8개짜리 판"이라고 생각하면 쉽다.** 전등 스위치 8개가 나란히 있고, 왼쪽부터 순서대로 128·64·32·16·8·4·2·1이라는 "무게"가 매겨져 있다고 하자(전부 2의 거듭제곱). 스위치를 어떻게 켜고 끄든 스위치 **개수는 항상 8개로 고정**이다 — 이게 "1바이트 = 8비트"의 실제 의미다.

`34`라는 값을 만들려면 `32`짜리 스위치와 `2`짜리 스위치만 켜면 된다(32+2=34). 나머지 6개는 꺼둔다.

| 스위치 무게 | 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1 |
|---|---|---|---|---|---|---|---|---|
| 34를 만들 때 | 0 | 0 | 1 | 0 | 0 | 0 | 1 | 0 |

32번과 2번 스위치만 ON이고, 이어서 쓰면 `00100010`(2진수 표기)이다.

여기서 헷갈리기 쉬운 지점 하나: `34`는 10진수로 **두 글자**라서 "8칸 중 2칸만 쓴 것" 처럼 보이기 쉽다. 하지만 그게 아니다 — 스위치는 항상 8개이고, `34`는 그중 2개(32번, 2번)만 켜져 있고 나머지 6개가 꺼져 있는 상태일 뿐이다. `255`를 만들려면 8개를 전부 켜야 한다(128+64+32+16+8+4+2+1=255) — 그래서 255가 1바이트로 표현 가능한 최댓값이다.

| 10진수 | 2진수(8비트) | 비고 |
|---|---|---|
| 0 | `00000000` | 전부 꺼짐 |
| 1 | `00000001` | 1번 스위치만 ON |
| 3 | `00000011` | 2번+1번 ON |
| 34 | `00100010` | 32번+2번 ON |
| 255 | `11111111` | 전부 ON, 1바이트 최댓값 |

즉 `34`, `1`, `255`는 전부 "스위치 8개짜리 상자" 하나에 들어있는 서로 다른 값일 뿐, **상자 크기(8개)는 절대 안 바뀐다.** 그 값을 `34`처럼 10진수 몇 글자로 적든 `00100010`처럼 2진수 8글자로 적든, 같은 값을 표기법만 다르게 쓴 것뿐이다. 배열 `[1, 0, 3, 34]`를 보고 "칸 수가 서로 다른가?" 싶을 수 있는데, 이 배열은 그냥 "바이트 4개, 각각의 값을 10진수로 표시한 것"이다 — 자릿수는 사람이 읽기 편하라고 생략된 것일 뿐, 실제로는 넷 다 8비트짜리 상자다.

## 엔디안 — 바이트를 나열하는 순서

숫자 하나가 255를 넘으면(예: GPS 좌표처럼 큰 정수) 바이트 하나로는 부족해서 여러 바이트를 이어붙여 표현해야 한다. 예를 들어 정수 `1`을 4바이트 정수로 표현하면 바이트 4개 중 3개는 `00`이고 나머지 1개만 `01`이다. 그런데 이 "01"을 4바이트 중 **어디에 놓을지**가 두 가지로 갈린다.

값 `1`을 4바이트 정수로 표현하면:

| 방식 | 바이트 순서 | 설명 |
|---|---|---|
| 빅 엔디안(big-endian) | `00 00 00 01` | 큰 자리(최상위 바이트)가 앞에 온다 |
| 리틀 엔디안(little-endian) | `01 00 00 00` | 작은 자리(최하위 바이트)가 앞에 온다 |

"엔디안"은 이 바이트들을 나열하는 순서 규칙이다. 사람이 숫자를 쓰는 방식(큰 자리가 왼쪽)과 같은 게 빅 엔디안, 반대가 리틀 엔디안이다. 어느 쪽이 맞다 틀리다의 문제가 아니라 **송신 쪽과 수신 쪽이 같은 규칙으로 합의해야만 하는** 문제다. 같은 4바이트를 받아도 어떤 순서로 읽느냐에 따라 완전히 다른 숫자가 나온다.

```js
const buffer = new Uint8Array([0x01, 0x00, 0x00, 0x00]).buffer;
const view = new DataView(buffer);

view.getInt32(0, true);  // 1          ← 리틀 엔디안으로 읽음
view.getInt32(0, false); // 16777216   ← 빅 엔디안으로 읽음 (같은 바이트인데 값이 다르다!)
```

`DataView`는 "이 오프셋부터 몇 바이트를, 부호 있게/없게, 어느 엔디안으로 읽을지"를 메서드 하나로 처리해주는 표준 API다. 자주 쓰는 메서드는 이렇다.

| 메서드 | 읽는 바이트 수 | 부호 | 비고 |
|---|---|---|---|
| `getUint8` / `getInt8` | 1바이트 | 없음/있음 | 1바이트는 순서 문제가 없어서 엔디안 인자가 없다 |
| `getUint16` / `getInt16` | 2바이트 | 없음/있음 | 두 번째 인자로 엔디안 지정 |
| `getUint32` / `getInt32` | 4바이트 | 없음/있음 | GPS 좌표처럼 큰 정수를 읽을 때 자주 쓰는 메서드 |
| `getFloat32` / `getFloat64` | 4·8바이트 | - | 소수점 있는 값을 그대로 실어 보낼 때 |
| `getBigInt64` / `getBigUint64` | 8바이트 | 있음/없음 | JS `number`로 못 담는 큰 정수, `BigInt`로 반환 |

이름의 규칙은 단순하다. `get` 뒤의 `Uint`/`Int`가 부호 없음/있음, 뒤에 붙은 숫자(`8`/`16`/`32`/`64`)가 비트 수(= 바이트 수 × 8)다.

`Uint`/`Int`가 자꾸 헷갈리면 이렇게 외우면 된다 — **"U"가 붙으면 Unsigned(부호 없음), U가 없으면 그냥 Int(부호 있음).**

| 표기 | 풀어 읽으면 | 부호 | 8비트 기준 범위 |
|---|---|---|---|
| `Uint8` | **U**nsigned Int8 | 없음 — 0 또는 양수만 | `0 ~ 255` |
| `Int8` | (Signed) Int8 | 있음 — 음수 포함 | `-128 ~ 127` |

`Uint16`/`Int16`, `Uint32`/`Int32`, `BigUint64`/`BigInt64`도 전부 이 규칙 그대로다 — **"U"의 유무만 보면 부호 유무가 바로 나온다.**

### 왜 `Int`가 `Uint`보다 최댓값이 절반인가

다시 스위치 비유로 보자. **부호 없음(`Uint`)** 은 스위치를 전부 "크기"를 만드는 데 쓴다. 1바이트(스위치 8개)면 2⁸=256가지 조합이 전부 크기 표현에 쓰여서 `0~255`가 나온다.

**부호 있음(`Int`)** 은 제일 왼쪽(최상위) 스위치 하나를 크기가 아니라 **"부호 표시(양수냐 음수냐)"** 로 따로 떼어 쓴다. 그럼 크기를 만드는 데 남는 스위치는 7개뿐이라, 조합은 2⁷=128가지로 줄어든다.

```
Uint8: 스위치 8개 전부 크기용        → 2⁸ = 256가지        → 0 ~ 255
Int8 : 스위치 7개는 크기, 1개는 부호  → 2⁷ = 128가지(양수쪽) → 0 ~ 127  (음수쪽에 다시 128가지: -128 ~ -1)
```

"최댓값이 절반"이라는 말은 이 뜻이다 — 스위치 하나를 부호 표시에 내주는 대신, 크기를 만들 수 있는 스위치가 하나 줄어서 조합 수가 정확히 절반(256 → 128)이 된다. 대신 그 절반만큼을 음수 쪽에 새로 배정해서, 전체 조합 수(256가지) 자체는 그대로 유지하면서 음수까지 표현할 수 있게 된다. 즉 "크기만 256가지"였던 걸 "양수 128가지 + 음수 128가지"로 나눠 쓰는 셈이다.

`Int32`/`Uint32`도 스위치 개수만 32개로 늘어날 뿐 원리는 같다.

```
Uint32: 스위치 32개 전부 크기용        → 2³²가지        → 0 ~ 4,294,967,295
Int32 : 스위치 31개는 크기, 1개는 부호  → 2³¹가지(양수쪽) → 0 ~ 2,147,483,647  (음수쪽 -2,147,483,648 ~ -1)
```

`Int32`의 최댓값(`2,147,483,647`)이 `Uint32`의 최댓값(`4,294,967,295`)의 거의 절반인 이유가 이거다 — 32개 스위치 중 1개를 부호에 내주고, 나머지 31개로만 크기를 만들기 때문이다. (음수를 실제로 어떤 비트 패턴으로 인코딩하는지는 "2의 보수" 방식인데, [바이너리 프로토콜 파싱](/articles/frontend/binary-protocol-parsing) 아티클의 CAN 신호 파싱 부분에서 실제 코드로 다룬다.)

2바이트 이상을 읽는 메서드는 전부 **두 번째 인자로 `littleEndian`(boolean)을 받는다.** `true`면 리틀 엔디안, `false`나 생략하면 빅 엔디안으로 읽는다.

## 바이트 하나에 여러 신호가 나눠 담기는 이유

여기까지는 "숫자 하나 = 바이트 몇 개"였다. 그런데 CAN 버스처럼 프레임 크기가 극단적으로 작은(최대 8바이트) 프로토콜에서는, 바이트 하나 전체를 신호 하나가 다 쓰지 않고 **비트 몇 개씩 쪼개서 여러 신호를 한 바이트에 같이 담는다.**

실제 예시(전기트랙터 CAN 데이터)로 보면, 바이트 하나(값 `116`, 2진수 `01110100`)를 8개 비트로 쪼개면 이렇다.

| 비트 위치 | 7 | 6 | 5 | 4 | 3 | 2 | 1 | 0 |
|---|---|---|---|---|---|---|---|---|
| 값 (116) | 0 | 1 | 1 | 1 | 0 | 1 | 0 | 0 |

그리고 이 8칸 안에 서로 다른 신호 4개가 겹쳐 있다.

| 비트 범위 | 신호명 | 비트 수 | 값 | 의미 |
|---|---|---|---|---|
| 7~5 | `PosnLvrSw` | 3비트 | `011` (10진수 3) | 레버 위치 3단계 |
| 4 | `RemoteCtrlValid` | 1비트 | `1` | 켜짐 |
| 3 | (미사용) | 1비트 | `0` | — |
| 2 | `PTOActSw` | 1비트 | `1` | 켜짐 |
| 1 | (미사용) | 1비트 | `0` | — |
| 0 | `BrkSeperatedSw` | 1비트 | `0` | 꺼짐 |

`BrkSeperatedSw`는 스위치를 **딱 1칸만** 배정받아서 켜짐/꺼짐 2가지(0 또는 1)밖에 표현 못 한다. `255`까지 표현하려면 스위치 8개를 전부 자기가 가져야 하는데, 나머지 7칸은 이미 다른 3개 신호가 나눠 쓰고 있어서 물리적으로 불가능하다. 이건 낭비가 아니라 반대다 — 켜짐/꺼짐만 필요한 신호에 8칸을 다 주면 7칸이 노니까, 필요한 만큼(1칸)만 주고 남는 칸을 다른 신호에 나눠준 것이다.

정리하면 **스위치(비트) 개수가 그 신호가 가질 수 있는 값의 가짓수를 정한다.**

| 배정받은 비트 수 | 표현 가능한 값 | 예시 |
|---|---|---|
| 1비트 | 2가지 (0~1) | 켜짐/꺼짐 스위치 |
| 3비트 | 8가지 (0~7) | 레버 위치 같은 작은 단계값 |
| 8비트(바이트 전체) | 256가지 (0~255) | 속도·전압처럼 넓게 변하는 값 |
