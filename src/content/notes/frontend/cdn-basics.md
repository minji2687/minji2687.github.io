> 초안. Lighthouse 노트에서 "로컬은 render-blocking 문제가 있는데 GitHub Pages 배포판은 괜찮다"는 걸 확인하다가 CDN 개념이 헷갈려서 따로 뺀 노트. 나중에 실제로 Cloudflare나 CloudFront 붙여보면서 캐시 히트/미스 스크린샷, `Cache-Control` 헤더 실험 추가할 것.

## CDN이 하는 일

CDN(Content Delivery Network)은 **오리진 서버(원본 파일이 있는 진짜 서버) 앞에, 전 세계 여러 지역에 흩어진 캐시 서버(엣지 서버)를 두는 것**이다. 사용자가 파일을 요청하면 오리진까지 안 가고, 사용자와 지리적으로 가까운 엣지 서버가 캐시된 복사본을 바로 돌려준다.

- 오리진 서버가 서울에 있고 사용자가 뉴욕에 있으면, 매번 태평양 건너 요청을 보내는 대신 뉴욕 근처 엣지 서버가 응답
- 오리진 서버의 부하도 줄어든다 — 같은 파일을 반복 요청하는 트래픽은 엣지에서 다 흡수하고, 캐시가 없을 때(cache miss)만 오리진까지 요청이 감

## 오해: CDN은 이미지 전용이 아니다

**"이미지 CDN"이라는 말 때문에 CDN = 이미지 최적화 서비스로 오해하기 쉬운데, 일반 CDN은 정적 파일이면 뭐든 캐싱한다** — HTML, CSS, JS, 폰트, 이미지, 동영상 전부.

두 가지를 구분해야 한다:

1. **일반 CDN**: 파일을 있는 그대로 엣지에 캐싱만 함 (Cloudflare, CloudFront, Fastly 등)
2. **이미지 전용 CDN**: 캐싱 + **이미지 리사이징/포맷 변환(WebP/AVIF 자동 변환)**까지 해주는 특화 서비스 (Cloudinary, imgix 등). 이건 CDN 기능 위에 이미지 가공 기능을 얹은 것이지, CDN 자체가 이미지만 다루는 게 아니다.

## 왜 어떤 배포는 설정 없이도 CDN이 붙어있나

[Lighthouse 노트](/notes/frontend/lighthouse-performance-metrics)에서 확인한 것처럼, 이 블로그(GitHub Pages 배포)는 로컬 dev 서버보다 배포판이 훨씬 빠르다. 이유는 **정적 호스팅 서비스 자체가 이미 CDN 위에서 동작하도록 설계돼 있기 때문**이다.

| 배포 방식 | CDN 자동 적용 | 비고 |
|---|---|---|
| GitHub Pages | O | 내부적으로 Fastly CDN 사용 |
| Vercel | O | 자체 엣지 네트워크 |
| Netlify | O | 자체 CDN |
| Cloudflare Pages | O | Cloudflare 엣지 네트워크 그 자체 |

이런 서비스들은 "정적 파일 올리면 알아서 전 세계에 뿌려준다"가 제품의 핵심 기능이라, 사용자가 CDN을 별도로 설정할 필요가 없다.

## 직접 서버를 운영하면 CDN이 없다

EC2, VPS(Lightsail, Droplet 등)에 nginx 같은 걸 직접 올려서 서버를 운영하면 얘기가 다르다. **서버가 딱 한 곳(리전 하나)에만 떠 있고, 전 세계 모든 요청이 그 서버 하나로 몰린다.** 사용자가 서버에서 멀수록 응답이 느려지고, 트래픽이 몰리면 서버 부하도 그대로 받는다. S3에 정적 파일만 올려놓는 것도 마찬가지 — S3 자체는 리전 하나에 있는 저장소라 CDN이 아니다.

이 경우 CDN을 쓰려면 **직접 붙여야 한다.**

## 자체 배포에 CDN을 붙일 때 쓰는 서비스

- **Cloudflare**: 개인 프로젝트에서 가장 흔한 선택. 도메인의 네임서버를 Cloudflare로 옮기고 프록시(주황 구름 아이콘)만 켜면, 별다른 설정 없이 정적 자원이 자동으로 엣지에 캐싱되기 시작한다. 무료 티어가 넉넉해서 개인 블로그/홈페이지엔 이걸로 충분한 경우가 많음.
- **AWS CloudFront**: S3에 정적 파일을 올려놓고, CloudFront에서 그 S3 버킷을 오리진으로 지정해 배포(distribution)를 만드는 조합이 전형적인 패턴. AWS 생태계 안에 있을 때 자연스러운 선택.
- **Fastly, Akamai**: 대규모 트래픽/엔터프라이즈에서 주로 씀. 세밀한 캐시 제어(VCL 스크립팅 등)가 가능하지만 개인 프로젝트엔 과함.
- **Bunny CDN**: 저렴하고 설정이 단순해서 개인 프로젝트에 인기 있는 대안.

정리하면, GitHub Pages처럼 "정적 호스팅 서비스"를 쓰면 CDN이 이미 포함된 상품을 쓰는 거고, EC2/VPS/S3처럼 "인프라 조각"만 쓰면 CDN은 별도 레고 블록으로 직접 조립해야 한다.

## S3 배포 vs EC2 배포 — CDN이 적용되는 범위가 다르다

**정적 사이트(S3)**: HTML/CSS/JS/이미지가 전부 S3라는 오리진 하나에 있으니, CloudFront를 그 앞에 붙이면 **전체가 통째로 CDN을 탄다.** 지금 이 블로그(GitHub Pages)와 구조적으로 동일 — 오리진 하나 + CDN 하나.

**서버사이드 앱(EC2)**: 동적 렌더링이 필요해서 S3만으로는 안 되는 경우. 이때도 CDN을 못 붙이는 게 아니라, **경로/파일 종류별로 캐싱 여부가 자동으로 갈린다.**

- **방법 1 — EC2 앞에 CDN 하나만 붙이기**: CloudFront나 Cloudflare를 EC2 앞에 그대로 둔다. CDN 입장에서 오리진이 S3든 EC2든 상관없다. 이미지/CSS/JS 같은 정적 파일 요청은 `Cache-Control` 헤더(또는 Cloudflare의 파일 확장자 기반 기본 캐싱 규칙)에 따라 CDN이 캐싱해서 응답하고, 로그인 페이지나 API 응답 같은 동적 HTML 요청은 캐싱 없이 매번 EC2까지 그대로 흘려보낸다. **CDN 하나가 요청 종류에 따라 캐싱할지 말지를 알아서 나눠 처리하는 것**이지, EC2 전체를 캐싱하거나 안 하거나의 이분법이 아니다.
- **방법 2 — 정적 파일만 S3로 분리 + CloudFront가 오리진 두 개를 경로별로 라우팅**: 이미지/빌드 산출물(JS/CSS)만 S3 버킷에 올리고, CloudFront 배포 하나에 오리진을 두 개 등록한다 — `/images/*`, `/static/*`은 S3로, 나머지 `/*`는 EC2(또는 ALB)로. 무거운 정적 파일은 S3+CDN 조합으로 오리진 부하 없이 서빙되고, EC2는 진짜 동적 로직만 처리한다. Rails/Django/Node 같은 서버사이드 앱에서 흔히 쓰는 실무 패턴.

> 아래 두 섹션(실무 CDN 적용 범위, CORS)은 아직 정리 중 — 개념은 맞는데 설명 순서·비유가 덜 다듬어졌음. 나중에 다시 손볼 것.

## 실무에서는 이미지(S3)만 CDN 붙이고 EC2는 그냥 둘까?

둘 다 있지만, 실서비스 규모라면 **EC2도 같은 CDN 밑에 같이 두는 방식(위의 방법 1/2)이 더 흔하다.** 이유는 캐싱 이득 때문만이 아니다.

- **작은 프로젝트**: `cdn.example.com`(CloudFront→S3)과 `www.example.com`(ALB→EC2)을 완전히 분리 도메인으로 따로 씀. 동적 응답은 캐싱해도 이득이 없으니 EC2는 CDN 없이 직접 노출. 단순함.
- **트래픽 있는 실서비스**: 도메인을 하나로 통일해서 `www.example.com` 밑에 정적/동적 요청을 다 두고, CDN이 경로별로 오리진만 나눠 라우팅. 캐싱 여부와 상관없이 **모든 요청이 CDN을 거친다** — DDoS 방어, TLS를 사용자와 가까운 엣지에서 처리(빠른 핸드셰이크), CDN이 오리진과 미리 열어둔 커넥션 재사용 같은 이득이 캐싱 없이도 생기기 때문.

### 도메인을 통일하면 왜 CORS를 안 신경 써도 되나

**CORS(Cross-Origin Resource Sharing)** 는 "브라우저가 웹페이지에게 다른 도메인의 데이터를 마음대로 읽지 못하게 막는 보안 규칙"이다. 비유하자면 — 웹페이지의 JS 코드가 다른 집(다른 도메인) 문을 두드려서 뭔가 가져오려고 할 때, 그 집 주인(서버)이 "너네 집(이 도메인)에서 온 요청은 들여보내도 돼"라고 미리 허가증(`Access-Control-Allow-Origin` 헤더)을 안 붙여주면, 브라우저가 알아서 차단해버린다.

여기서 "다른 도메인"의 기준은 생각보다 엄격하다 — `www.example.com`과 `cdn.example.com`은 같은 회사가 운영해도, 브라우저 눈에는 **서브도메인이 다르면 완전히 다른 출처(origin)** 다.

- 이미지를 `<img src="https://cdn.example.com/logo.png">`처럼 그냥 화면에 보여주기만 할 땐 사실 CORS가 안 걸린다.
- 하지만 프론트엔드 JS가 `fetch()`로 그 도메인의 데이터를 읽어와야 하거나(API 호출), 웹폰트(`@font-face`)를 다른 도메인에서 불러오거나, `<canvas>`에 그린 이미지를 코드로 읽어야 하는 경우엔 **브라우저가 CORS 헤더를 요구한다.** 이때 S3/CDN 쪽에 CORS 설정(허용할 도메인 목록 등)을 별도로 해줘야 한다.
- 반면 정적 파일과 API를 전부 `www.example.com` 하나의 도메인 아래(`/images/*`, `/api/*`처럼 경로만 다르게) 두면, 브라우저 입장에서는 전부 "같은 집"이라 애초에 이 허가증 절차 자체가 필요 없다.

즉 도메인을 나누면(예: 정적 파일은 `cdn.example.com`, API는 `www.example.com`) 매번 CORS 허용 설정을 관리해야 하는 번거로움이 생기고, 하나로 합치면 그 번거로움 자체가 사라진다는 뜻이다.

## 캐시 동작을 제어하는 법

CDN이 파일을 얼마나 오래 캐싱할지는 오리진 서버가 응답에 실어 보내는 `Cache-Control` 헤더로 정한다.

- `Cache-Control: public, max-age=31536000, immutable` — 1년간 캐싱, 파일이 안 바뀐다고 보장(주로 해시가 파일명에 붙는 빌드 산출물에 씀)
- `Cache-Control: no-cache` — 매번 오리진에 재검증(revalidate) 요청
- CDN 서비스 대시보드에서 강제로 캐시를 비우는 "purge/invalidate" 기능도 따로 있음 — 배포 후 옛날 파일이 계속 캐시에 남아있는 문제를 해결할 때 씀
