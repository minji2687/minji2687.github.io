## 할루시네이션이 발생하는 이유

> 학습한 데이터를 바탕으로 **확률을 기반으로 단어를 예측**하기 때문이다.

예를 들어 "그"라는 단어 다음에 올 단어를 예측한다고 하면, 모델은 다음과 같은 확률 트리를 따라 단어를 이어붙인다.

```
그 ─ 집(0.4) ─ 사(0.1) / 은(0.2) / 에(0.7)
  ─ 책(0.5) ─ 이(0.4) / 을(0.3) / 읽(0.3)
  ─ 사람(0.1) ─ 에게(0.3) / 처럼(0.5) / 누구(0.2)
```

이 트리에서 확률이 높은 경로를 따라가면 "그 집에", "그 책이", "그 사람처럼" 같은 문장이 만들어진다.

즉 모델은 "사실"을 검색해서 답하는 게 아니라, 학습 데이터에서 **통계적으로 그럴듯한 다음 단어**를 이어붙일 뿐이다. 그 결과 그럴듯하지만 사실과 다른 답(할루시네이션)이 나올 수 있다 — 그래서 모델이 "추측"하는 대신 **실제 출처를 찾아보고 답하게** 만드는 RAG가 필요해진다.

---

## RAG란 무엇인가 (검색증강생성, Retrieval-Augmented Generation)

> 답변의 지식 출처를 제공하여, 해당 출처를 검색한 내용을 기반으로 답변을 제공한다 — **넓은 의미의 Context Providing**.

![RAG 아키텍처 흐름](/images/rag/rag-architecture-overview.png)

핵심은 LLM에 질문만 던지는 게 아니라, **질문과 관련된 정보를 먼저 검색해서 "강화된 컨텍스트(Enhanced Context)"로 만들어 함께 전달**한다는 것. 모델은 자기 확률 분포에만 의존하지 않고, 검색해 온 출처를 근거로 답을 생성하게 된다.

---

## RAG 진행 과정

1. 사용자의 질문 입력받기
2. 질문을 **임베딩 과정**을 통해 벡터로 변환
3. 변환된 벡터를 기반으로 벡터 DB에서 유사도를 바탕으로 단어들을 검색
4. 검색된 정보들을 Context로 하여 LLM에 입력
5. 최종 답변 추출

2번과 3번에서 "임베딩"이 핵심이다 — 출처 문서와 질문을 **같은 벡터 공간**에 올려놓아야 유사도 비교가 가능하기 때문.

---

## 인간의 인지 과정에 빗댄 비유

RAG의 흐름은 사람이 응대하는 과정과 닮아 있다.

1. 고객의 요청 이해하기 (ex. 환불)
2. 고객 응대 매뉴얼 참고 — 상대방의 질문과 연관된 페이지 찾기
3. 연관된 페이지 정보를 바탕으로 고객에게 답변

| 사람의 응대 | RAG 아키텍처 |
|---|---|
| 고객의 요청 이해 | Prompt + Query |
| 매뉴얼에서 연관 페이지 찾기 | Search Relevant Information → Knowledge Sources |
| 찾은 페이지 정보 | Relevant Information / Enhanced Context |
| 그 정보를 바탕으로 답변 | LLM이 Enhanced Context 기반으로 답 생성 |

사람이 기억에만 의존해 즉흥적으로 답하지 않고 매뉴얼에서 근거를 찾아 답하듯, RAG도 모델의 "기억(확률 분포)"이 아니라 외부 지식에서 찾은 근거를 바탕으로 답하게 만드는 구조다.

## 벡터와 임베딩

> 핵심: 데이터는 벡터로 저장된다.

RAG가 "관련 정보를 검색"하려면, 그 대상이 되는 지식 출처(문서, DB 등)가 **벡터(vector)** 형태로 저장되어 있어야 한다. 질문도 벡터로 변환한 뒤, 가장 가까운(유사한) 벡터를 찾는 방식으로 검색이 이루어진다 — 그래서 "벡터"와 "임베딩"이 다음 핵심 키워드가 된다.

### 벡터란?

> 크기와 방향을 가지는 물리량 — 쉽게 말하면 그냥 좌표.

![벡터는 곧 좌표](/images/rag/rag-vector-concept.png)

화살표(방향 + 크기)로 그린 벡터와 `[x, y]` 형태의 좌표값은 같은 것을 가리키는 **서로 호환 가능한 표현**이다. 위 그림의 `[2, 1]`, `[-1.5, 0.5]`, `[0.1, -3]`은 모두 원점 `(0, 0)`에서 뻗어나간 화살표이자, 동시에 좌표값으로도 표현 가능하다.

### 임베딩이란?

> 컴퓨터가 알아들을 수 있게, 인간의 문자열을 실수 형태인 벡터로 표현한 결과.

![임베딩 모델의 동작 방식](/images/rag/rag-embedding-concept.png)

"꽁꽁...", "그럼 제가...", "난 대학시절..." 같은 문자열을 **Embedding model**에 통과시키면 `[0.7 | 0.24 | ... | -0.4 | 0.12]`처럼 실수값이 나열된 배열(벡터)이 나온다. 즉 임베딩은 사람의 언어를, 컴퓨터가 거리·유사도를 계산할 수 있는 좌표(벡터)로 바꾸는 과정이다.

### 문자의 벡터화 (실제 예시)

![단어들을 벡터 공간에 투영한 예시](/images/rag/rag-word-vector-projection.png)

실제로 단어들을 임베딩해서 벡터 공간에 투영해보면, **의미가 비슷한 단어들끼리 가까운 위치에 모인다** (위 예시에서 `professors`와 의미적으로 연관된 단어들이 한 군데에 군집을 이루는 것처럼). RAG의 벡터 검색은 결국 이 원리를 이용해 "질문과 의미적으로 가까운 문서"를 찾아내는 것이다.

→ 이게 곧 **벡터 검색(redis-stack 등)**으로 이어지는 핵심 연결고리.
