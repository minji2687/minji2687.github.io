## 배경

긴트는 트랙터·이앙기에 자율주행 소프트웨어를 얹는 회사다. 예전에는 설치 정보를 엑셀로 관리했다. 고객에게 설치됐다고 기록된 제품과 실제 설치된 단말기 정보가 어긋나는 경우가 잦았고, 이 미스매치 때문에 운영 관계자 전체가 힘들어했다. 주 제품뿐 아니라 거기 딸린 부품들도 각각 제품번호가 있어서 정확한 히스토리 관리가 필요했는데, 그전까지는 그런 체계 자체가 없어서 문제가 생길 때마다 원인을 추적하기 힘들었다.

그래서 고객 가입부터 제품 설치까지 필수 정보 입력을 의무화하기로 했다. 하드웨어 키트를 현장에 설치하는 기사가 종이 매뉴얼과 엑셀로 작업하던 과정을 앱으로 디지털화하면서, 차량 정보·치수 같은 필수 항목을 입력하지 않으면 제품을 개통할 수 없도록 막았다. 이렇게 쌓인 설치 데이터를 조회·수정하는 어드민도 별도로 구축했다.

설치 흐름은 7단계로 고정돼 있다. 설치 정보 확인 → 사전 점검 → 구성품 등록 → 장비 등록(차량 치수·마운팅 위치·IMU 캘리브레이션) → 계정 생성 → 완료. 현장에서 스마트폰으로 쓰는 앱이라 모바일 우선 설계가 기본 전제였다.

비즈니스 플랫폼팀에서 프론트엔드 개발을 담당했다. 백엔드 개발자 1명, 기획자 1명과 협업했다.

## 1. 바코드 스캐너 라이브러리 검증

구성품(인터페이스박스·원터치스위치·스티어링휠)의 시리얼 넘버를 바코드로 등록한다. 기사가 현장에서 스마트폰 카메라로 찍는 흐름이라 인식 속도와 정확도가 실용성의 핵심이었다.

문제는 하드웨어 자체에 있었다. 제품에 붙은 바코드는 원래 현장 스캔을 상정하고 만든 게 아니었다. 내부 재고 관리용으로 형식상 찍어둔 것이라 크기가 손가락 한 마디 수준으로 작았다. 이 앱을 만들면서 처음으로 그 바코드를 현장에서 실제로 인식해야 하는 상황이 됐다.

상용 라이브러리인 Dynamsoft는 인식률이 압도적이었다. 하지만 연간 라이선스 비용이 500만 원 수준이라 현실적으로 선택할 수 없었다. 무료 오픈소스 라이브러리들로 해결해야 했다.

무료 라이브러리들도 일반 크기의 바코드는 어느 정도 인식했다. 문제는 초소형 바코드였다. `@zxing/browser`, `quagga`, `window.BarcodeDetector` 등을 실시간 스트리밍 방식으로 구현해봤지만, 손가락 한 마디 크기의 바코드를 라이브 영상에서 인식하는 건 무료 라이브러리 수준에서는 신뢰할 수 없었다.

방향을 바꿨다. 실시간 인식을 포기하고, 사진을 찍어서 정지 이미지를 분석하는 방식으로 전환했다. 정지 이미지는 고해상도로 캡처할 수 있고, ROI(관심 영역) 크롭과 줌을 조합하면 작은 바코드도 충분히 해상도를 확보할 수 있었다. `@zxing/browser`, `quagga`, `window.BarcodeDetector` 등 10개 이상의 구현체를 BarcodeScanner부터 BarcodeScanner10까지 번호를 붙여 독립 컴포넌트로 만들고 직접 비교했다.

인식 엔진은 브라우저 환경에 따라 분기했다. 최신 Chrome처럼 `window.BarcodeDetector`를 지원하는 환경에서는 그것을 우선 사용했다. 브라우저가 직접 제공하는 가장 로우레벨의 API라 무료 라이브러리 중에서 인식률이 가장 좋았다. iOS Safari나 구버전 Chrome처럼 지원되지 않는 환경은 QuaggaJS로 폴백했다. 앱이 아닌 브라우저 기반이다 보니 어떤 방식도 인식률이 완벽하지는 않았지만, 이 조합이 현실적으로 가장 나은 선택이었다.

UX도 기획자와 함께 많이 고민했다. 구성품을 4개 등록해야 하는데, 매번 "바코드를 크게 찍으세요"라는 안내를 반복하면 현장 기사 입장에서 답답하다. 그래서 세션 내 첫 번째 스캔 시에만 안내 모달을 띄우도록 했다. `hasShownGuideInSession` 플래그로 세션당 한 번만 노출하고, 이후 3개는 바로 카메라로 진입한다. 모달에서 "스캔 시작"을 누르면 `input[capture="environment"]`로 카메라가 바로 열리는 구조다.

## 2. 차량 타입별 동적 폼

트랙터와 이앙기는 자율주행에 필요한 측정값이 다르다. 항목 수도, 항목 종류도 다르다. 이걸 폼 컴포넌트를 따로 만들지 않고 `DimensionForm` 하나로 공유했다.

각 측정 항목을 탭 하나로 보고, 탭마다 라벨·설명·이미지 경로를 담은 객체(`DimensionTab`)를 만들었다. 차량 타입별로 이 탭 객체 배열만 다르게 구성해서 넘긴다. 폼은 가로로 스와이프해서 탭을 넘기는데, 탭이 바뀌면 입력 필드뿐 아니라 안내 이미지도 같이 바뀐다. 탭 하나에 입력값과 그 값을 설명하는 이미지를 묶어뒀기 때문에 가능한 구조다.

값을 읽고 쓰는 로직(`getCurrentValue`, `onValueChange`)과 검증 로직(`getCurrentError`, `getCurrentErrorMessage`)은 폼 내부에 두지 않고 전부 외부에서 함수로 주입받는다. `DimensionForm`은 그 자체로는 어떤 차량 타입인지, 어떤 필드가 몇 번 탭인지 전혀 모른다. 트랙터/이앙기 각각의 페이지가 자기 매핑과 검증 규칙을 들고 와서 꽂아주는 식이다. 그래서 차량 타입이 늘어나도 폼 자체는 손대지 않고 탭 배열과 주입 함수만 추가하면 된다.

마운팅 위치(높이·횡방향·종방향)와 IMU 초기값(피치·롤·요)도 같은 패턴으로 별도 단계에서 입력받는다. 측정값이 자율주행 정확도에 직접 영향을 주는 데이터라 모든 수치 입력에 검증을 붙였다.

## 3. 서버 상태와 동기화되는 설치 진행 상태

설치 흐름이 7단계로 길다 보니, 기사가 작업 중간에 이탈하는 경우가 종종 생겼다. 인증키로 재접속했을 때 처음부터 다시 입력하게 하면 현장에서 쓸 수 없는 앱이 된다. 이전까지 입력한 데이터는 살리고, 입력하지 않은 지점부터 이어서 진행하도록 만들어야 했다.

이걸 위해 단계마다 데이터를 저장할 때 `installProcess` 값(10·20·30·…·80)을 같이 서버로 보낸다. 단순히 순서대로 진행한다고 가정할 수 없었던 이유는 각 단계에 뒤로가기 버튼이 있어서다. 기사가 임의로 이전 단계로 돌아가 값을 고칠 수 있기 때문에, 지금 저장하는 데이터가 전체 흐름에서 몇 번째 단계의 데이터인지를 매 저장마다 명시적으로 알려줘야 했다.

재접속 시에는 인증키 확인 API 응답에 서버가 마지막으로 저장된 `installProcess`를 내려준다. 앱은 이 값을 기준으로 `INSTALL_PROCESS_STEPS` 배열에서 다음 단계를 찾고, `INSTALL_PROCESS_ROUTES` 맵으로 해당 경로에 바로 이동시킨다. 이동 전에는 `initializeFromInstallResult`로 그동안 입력했던 전체 데이터를 Zustand 스토어에 복원한다. 이때 서버 응답에 값이 없는 필드까지 덮어써서 로컬 상태를 날리는 걸 막기 위해 `safeUpdate` 헬퍼를 만들어 썼다. 서버 값이 있을 때만 갈아끼우고, 없으면 기존 로컬 값을 그대로 둔다.

## 4. 국제화 (next-intl)

한국 기사만 쓰는 앱이 아니라 해외 현장 대응도 요구사항에 있었다. `next-intl`을 선택한 이유는 Next.js의 서버 사이드(App Router, 미들웨어)에서 그대로 동작한다는 점이었다. 클라이언트에서만 번역을 갈아끼우는 라이브러리였다면 서버 컴포넌트와 미들웨어 단계에서 로케일을 처리할 수 없었을 것이다.

App Router 기반이라 페이지 자체가 서버에서 렌더링되는데, 로케일이 클라이언트에서만 결정되는 라이브러리였다면 서버는 어떤 언어로 렌더링해야 할지 모른 채 일단 기본 언어로 그려야 했을 것이다. 미들웨어가 요청 단계에서 먼저 로케일을 정해주면 서버 컴포넌트가 그 로케일에 맞는 메시지 파일을 바로 가져와 처음부터 맞는 언어로 렌더링할 수 있다.

라우팅은 `[locale]` 세그먼트로 구성하고, 미들웨어에 `next-intl`의 `createMiddleware(routing)`를 한 줄로 꽂아서 요청이 들어올 때마다 로케일을 감지하고 올바른 경로로 리다이렉트하게 했다. 지원 언어(`ko`, `jp`)와 기본 로케일, URL에 로케일을 항상 붙일지 같은 설정은 `routing.ts` 한 곳에서 관리한다.

번역 텍스트는 언어별로 JSON 파일 하나씩 나눠 관리한다(`messages/ko.json`, `messages/jp.json`). 요청이 들어오면 그 요청의 로케일에 맞는 파일만 동적으로 불러와서 서버 컴포넌트에 내려준다. 모든 언어의 메시지를 한 번에 들고 있지 않고 필요한 언어 파일만 읽는 구조다.

페이지 이동에는 `next/navigation`의 기본 `useRouter`, `Link`를 쓰지 않는다. `next-intl`이 `createNavigation`으로 만들어주는 로케일 인식 버전의 `useRouter`/`Link`/`usePathname`을 따로 만들어 전역에서 그것만 쓰도록 했다. 기본 라우터를 쓰면 페이지를 이동할 때 URL의 로케일 세그먼트가 유지되지 않고 깨질 수 있어서다. 전환해도 현재 단계와 입력 상태는 그대로 유지된다.

## 5. AWS Amplify CI/CD

브랜치 푸시 → 빌드 → 배포로 이어지는 파이프라인을 Amplify 콘솔에서 직접 구성했다. 환경변수도 콘솔에서 관리한다.

배포 환경은 브랜치로 분리했다. `develop` 브랜치에 푸시하면 dev 환경으로, `main` 브랜치에 푸시하면 prod 환경으로 각각 자동 배포된다. 기능 개발 중에는 dev 환경에서 자유롭게 확인하고, 검증이 끝난 뒤에만 `main`에 머지해 운영 환경에 반영하는 흐름이다.
